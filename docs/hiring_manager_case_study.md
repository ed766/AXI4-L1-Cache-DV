# Hiring-Manager Case Study: Early `WLAST` Detection

## Design Intent

The cache writes back a dirty 32-byte victim line as four 64-bit AXI beats. `WLAST` must be asserted only on beat three, after all four beats have been presented.

## Injected Failure

The `CACHE_BUG_WLAST_EARLY` mutation changes the writeback termination condition so `WLAST` is asserted on beat two. This models a realistic control bug where the memory system could accept an incomplete dirty-line writeback.

## Symptom

The `dirty_evict` scenario creates a modified victim and then forces replacement. The writeback begins correctly, but the mutated DUT terminates the burst one beat early.

## Detection

The named assertion `a_wlast_exactly_final_beat` catches the failure before the burst can be treated as a legal line writeback. The same failure class is also protected by the C++ trace replay and final backing-memory comparison.

## Root Cause Model

The injected mutation compares the writeback beat counter against `2` instead of the architectural final beat `3`. The nominal RTL uses the final-beat condition and passes directed regression, mutation validation, formal checks, and C++ trace replay.

This case demonstrates the intended verification flow: inject a realistic control bug, catch it with a named protocol assertion, confirm downstream scoreboard protection, and keep a reproducible waveform/debug artifact.
